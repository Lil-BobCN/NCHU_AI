# SDAR-0004: Phase 1 前端视觉方向与设计 Token

## 1. 一句话结论

推荐采用“正式可信的高校数字服务 + 现代科技感 + AI 助手 + 桌面工作台”的视觉方向。

具体执行上：

- 主站首页和学生端：使用 shadcn/ui 或自定义样式做更现代、更有产品感、更有轻科技感的体验。
- 辅导员端和管理后台：使用 Ant Design 的工作台能力，保证表格、筛选、表单、菜单、状态反馈成熟稳定。
- 全局设计 token 使用 CSS variables 作为源头，再映射到 Ant Design `ConfigProvider` 和 shadcn/ui theme tokens，避免两套组件库风格割裂。
- Phase 1 只确定视觉规则和 skeleton 设计边界，不创建前端工程，不写页面代码。

### 2.1 视觉基调

**原问题**

是否同意 Phase 1 采用“正式可信的高校数字服务 + 现代 AI 助手 + 桌面工作台”的视觉基调？

> **产品经理回复**
>
> 我同意你的初始建议，但希望视觉上能有现代感和科技感，不要过于传统或过于严肃。（我会后续导入skills，帮助你不仅能在网络上搜寻资源转化为视觉设计，并且能有效地通过提示词让你明确设计风格）

**Codex 回应**

接受这个修正。后续视觉基调不再写成偏保守的“正式后台感”，而是调整为：

- `正式可信`：保证学校、辅导员、管理者看起来可靠。
- `现代科技感`：首页、登录页、学生端要有 AI 产品气质，界面要干净、精致、有当代软件感。
- `不过度严肃`：避免传统政务系统、老式后台、纯表格系统的沉闷感。
- `不过度炫技`：不使用大面积深色科技风、紫色渐变、光斑和装饰圆球来制造“AI 感”。

处理结论：SDAR-0004 的主视觉方向修订为“正式可信 + 现代科技感 + AI 助手 + 桌面工作台”。后续如果你导入视觉设计 skills，我会把它作为风格输入，但仍然受本审批包的边界约束。

**Codex 初始建议**

推荐该方向。

原因：

- 这个 Demo 面向学校场景，需要可信、克制、正式，不能像娱乐网站、营销落地页或临时后台。
- 同时它又是 AI 辅导员产品，需要比传统后台更现代、更有产品感。
- 学生端要亲和、清晰；辅导员端和管理后台要高效、可扫描。

### 2.2 主色与辅助色

**原问题**

是否同意使用“中性浅色底 + 稳定蓝色主色 + 青绿色辅助色 + 语义状态色”的配色方向？

> **产品经理回复**
>
> 我同意你的初始建议，但希望视觉上能有现代感和科技感，不要过于传统或过于严肃。

**Codex 回应**

接受。颜色策略会保留蓝色和青绿色，但视觉呈现不做“传统蓝白后台”。具体调整为：

- 主色蓝用于可信、导航、主操作，不大面积铺满页面。
- 青绿色用于 AI 辅助、来源引用、知识命中等轻科技感提示。
- 背景仍以浅色为主，但可以通过细腻边框、轻量阴影、局部信息面板、精致图标和页面层级体现现代感。
- 不把科技感等同于深色背景、霓虹渐变或紫色光效。

处理结论：配色方向保留，但执行目标改为“浅色现代科技产品”，不是“传统学校后台”。

**Codex 初始建议**

推荐该方向。

初始色彩策略：

- 背景以浅灰白为主，避免压抑。
- 主色使用稳定蓝色，用于主按钮、当前导航、关键操作。
- 辅助色使用青绿色，用于 AI 辅助、来源引用、成功状态以外的柔和强调。
- 警告、错误、成功使用明确语义色。
- 避免大面积紫色、深蓝、渐变、装饰光斑，防止页面变成单一科技风或营销风。

### 2.3 首页视觉结构

**原问题**

是否同意主站首页采用“产品型顶部导航 + 正式首屏 + 两个角色入口卡片”的结构？

> **产品经理回复**
>
> 我需要你跟我以用户的角度，详细描述一下这个结构是什么样子的，为什么它能满足用户需求，以及它的优缺点。你也可以提供一些参考链接或者示例来帮助我更好地理解这个结构。
> 最重要的是给我描述出来用户是怎么交互的

**Codex 回应**

该项已被 SDAR-0005 修订。

最新处理结论：主站首页不再采用“两个角色入口卡片”作为主结构，而是采用参考 Linear 类产品官网的信息结构：

1. 用户从学校官网链接或 Demo URL 进入主站首页。
2. 顶部导航展示产品能力、使用流程、适用角色、资源/文档和登录入口。
3. 首屏先表达产品定位：这是面向学校场景的 AI 辅导员工作平台。
4. 页面中部展示产品特性，例如学生咨询、辅导员案例工作台、知识资源与统计。
5. 用户点击顶部“登录”或首屏“登录体验”后进入 `/login`。
6. 登录页再展示学生和老师/辅导员两类角色选项，并根据角色展示不同欢迎语和引导文案。
7. 管理后台仍通过独立链接进入，不作为主站公开角色入口。

为什么这样满足用户需求：

- 主站更像正式产品，而不是简单角色跳转页。
- 用户先理解产品能做什么，再进入登录，可信度更高。
- 角色选择集中在登录页，登录文案和跳转目标更清楚。
- 保留管理后台独立入口，权限边界更干净。

优点：

- 更符合成熟产品官网的信息节奏。
- 可以展示产品特性和价值，不只展示入口。
- 后续视觉上更容易做出现代产品感。
- 登录页承担角色选择，交互边界更清晰。

缺点：

- 首页规划比简单入口页更复杂。
- 需要避免做成过度营销页。
- 登录页必须设计清楚，否则用户可能不知道在哪里选择角色。

参考依据：本结论以已批准的 `SDAR-0005` 和 `.omx/prototypes/frontend-interaction-flow-demo.html` 为准；Linear 只作为产品官网信息结构参考，不复制品牌、文案或视觉细节。

**Codex 初始建议（已被 SDAR-0005 取代）**

原初始建议曾把首页设计为老师/辅导员与学生两个入口的轻量跳转页。该建议已经被产品经理后续反馈和 `SDAR-0005` 修订，不再作为后续实现依据。

当前有效结论是：

- 首页负责产品定位、能力展示、可信度建立和登录入口。
- `/login` 负责学生与老师/辅导员的正式角色选择。
- 管理后台仍通过独立链接进入，不在主站公开展示。
- 当前低保真原型只用于确认结构和交互顺序，不锁定最终视觉样式。

**原问题**

是否同意辅导员端和管理后台采用“顶部栏 + 左侧菜单 + 主内容区”的工作台视觉结构，并优先参考 Ant Design / Ant Design Pro？

> **产品经理回复**
>
> 我同意你的初始建议

**Codex 回应**

确认采纳。辅导员端和管理后台属于高频工作台，不适合做成首页式大卡片或聊天页式结构。后续 skeleton 里会按“顶部栏 + 左侧菜单 + 主内容区”规划：

- 顶部栏负责当前系统、角色、账号和全局状态。
- 左侧菜单负责角色内功能导航。
- 主内容区负责列表、详情、筛选、表单、统计和审计。

处理结论：该项无修改要求，继续沿用 Ant Design / Ant Design Pro 的成熟工作台模式。

推荐该方向。

原因：

- Ant Design 的 `Layout` 与主题 token 能覆盖成熟后台布局。
- 辅导员端和管理后台需要表格、筛选、详情、状态、活动记录，工作台型布局比营销式布局更合适。
- 该结构与 SDAR-0003 已批准的导航方案一致。

### 2.5 学生端视觉结构

**原问题**

是否同意学生端采用“清晰问答主区 + 来源卡片 + 历史会话入口”的轻工作台结构，而不是复杂后台结构？

> **产品经理回复**
>
> 我同意你的初始建议，但希望视觉上能有现代感和科技感，不要过于传统或过于严肃。但是我需要你更详细地描述一下这个结构是什么样子的，为什么它能满足用户需求，以及它的优缺点。你也可以提供一些参考链接或者示例来帮助我更好地理解这个结构。最重要的是给我描述出来用户是怎么交互的。

**Codex 回应**

接受。学生端不是后台管理系统，它的核心任务是“快速提问、看懂回答、知道来源、能继续追问”。

用户交互流程如下：

1. 学生登录后进入 `/student`。
2. 页面首先展示一个清晰的问答主区，中心是问题输入框，旁边或下方有推荐问题、常见问题或历史入口。
3. 页面持续显示 Demo/模拟数据标识，避免学生误以为系统接入了真实个人档案。
4. 学生输入问题并提交。
5. 回答区域展示：
   - 回答正文。
   - 来源/资源卡片。
   - 如果问题不在 Demo 覆盖范围内，展示明确兜底回答。
6. 学生可以点击来源卡片查看资源详情。
7. 学生可以打开历史会话，继续查看或接着提问。

为什么这样满足学生需求：

- 学生不需要管理数据，只需要低阻力地提问。
- 来源卡片让回答更可信，不像“凭空生成”。
- 历史会话支持连续使用，但不会让页面变成复杂后台。
- 这个结构能自然承接 Phase 3 的确定性答案来源适配器，也能承接 Phase 7 之后的 RAG 来源引用。

优点：

- 聚焦问答，学习成本低。
- 比传统后台更现代，更像 AI 助手产品。
- 来源卡片增强可信度。
- 后续接入真实 API/RAG 时不需要推翻页面结构。

缺点：

- 如果信息太少，可能显得功能单薄，所以需要推荐问题、资源入口和历史入口补足。
- 如果信息太多，学生端会变成后台，所以信息密度要控制。
- loading、empty、fallback、错误状态后续必须认真做，否则交互会显得不完整。

处理结论：学生端继续采用“轻工作台 + AI 问答主区”，视觉上加强现代科技感，但不做复杂后台。

**Codex 初始建议**

推荐该方向。

学生端重点是能快速提问、看懂答案、看到来源，不应像管理后台一样复杂。

推荐结构：

- 中心区域：问答输入和回答。
- 侧边或局部区域：来源卡片、资源入口、历史会话。
- 明确标注 Demo/模拟数据。

### 2.6 设计 Token 统一方式

**原问题**

是否同意以 CSS variables 作为设计 token 源头，再映射到 Ant Design 和 shadcn/ui？

> **产品经理回复**
>
> 这部分技术我不懂，详细描述出为什么要这么做，以及它的优缺点。我需要你用非技术的语言来解释这个问题，这样我才能更好地理解你的建议。（什么需求促使你觉得有必要做这项技术？；有哪些解决方案？；你为什么选择当前技术方案？）

新回复：
shadcn/ui有对应的skills帮助你更好的使用这套组件，你自己配置上这个skills，在开发的时候使用

**Codex 回应**

接受。这里用非技术语言解释。

设计 token 可以理解为“产品的统一视觉规则”。它像学校制服标准或品牌规范：

- 主按钮蓝色永远是同一种蓝。
- 卡片圆角永远遵循同一套标准。
- 页面之间的间距有统一节奏。
- 警告、错误、成功这些颜色在全系统里含义一致。

为什么本项目需要它：

我们已经批准使用两类前端 UI 来源：

- Ant Design：适合辅导员端、管理后台这种工作台。
- shadcn/ui 或自定义样式：适合首页、登录页、学生端这种更现代、更产品化的界面。

这能兼顾开发效率和好看程度，但也带来一个风险：如果没有统一规则，首页可能像一个产品官网，后台像另一个管理系统，学生端又像第三种风格，整体会像“拼起来的三个产品”。

可选方案：

| 方案 | 含义 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 不做统一 token | 每个页面自己调样式 | 一开始最快 | 很容易风格不一致，后期难维护 |
| 只用 Ant Design 主题 | 所有页面尽量跟 Ant Design | 工作台稳定 | 首页、学生端会太像后台，不够好看 |
| CSS variables 作为统一源头 | 先定义一套视觉规则，再映射给 Ant Design 和 shadcn/ui | 统一、灵活，适合两套组件共存 | 需要实现时严格遵守规则 |

为什么选择 CSS variables：

- Ant Design 支持主题 token。
- shadcn/ui 官方也推荐用 CSS variables 做主题。
- 它可以作为两套组件之间的“共同语言”：同一个蓝色、同一个圆角、同一套间距，可以同时影响工作台和产品页面。

处理结论：继续推荐 CSS variables 作为设计 token 源头。它不是为了炫技，而是为了保证整个 Demo 看起来像一个统一产品。

**Codex 初始建议**

原因：

- Ant Design 官方支持通过 Design Token 自定义主题。
- shadcn/ui 官方推荐使用 CSS variables 做主题。
- 我们同时使用 Ant Design 和 shadcn/ui，必须用统一 token 控制颜色、字号、间距、圆角和阴影，否则页面会像两个产品拼在一起。

### 2.7 圆角、密度与风格边界

**Codex 回应**

`.omx/specs/autoresearch-frontend-visual-design-systems/report.md`

本次参考的成熟设计系统包括：

- Ant Design：确认工作台适合使用成熟企业 UI 模式，且默认基础圆角为 `6px`。
- shadcn/ui：确认首页、登录页、学生端可使用 CSS variables 做灵活主题。

- 工作台使用中高信息密度。
- 首页保持更宽松的留白。
- 学生端使用中等密度，保证可读，不做后台式拥挤布局。

处理结论：你提出的“用成熟设计系统指导决策”已纳入本审批包，后续设计 token 以该 autoresearch 报告作为依据。

**Codex 初始建议**

推荐该方向。

原因：

- 8px 以内圆角更符合正式工具产品，不会过度卡通化。
- 辅导员端和管理后台需要高效扫描，不能做成过大的营销卡片。
- 首页和学生端可以适当留白，提高第一印象和可读性。

- `/login` 承担学生与老师/辅导员的正式角色选择、欢迎语和引导文案。

- Demo/模拟数据边界必须持续可见。

允许后续微调的内容是：

- 首页视觉密度、区块节奏、首屏布局和产品特性展示方式。
- 登录页角色选择卡片的样式、尺寸、图标和文案层级。
- 学生端、辅导员端、管理后台的局部留白、色彩强度、组件组合和交互状态。
- 具体 token 值，例如主色深浅、卡片阴影、间距比例和字体层级。

处理结论：SDAR-0004 可以继续作为视觉方向与 design token 审批包推进，但当前原型不会被写死为最终视觉标准。

## 3. 已批准上游决策

- `SDAR-0002`：前端技术栈为 React + TypeScript + Vite + Ant Design / shadcn-ui。
- `SDAR-0003`：独立主站 + 独立管理后台 + 桌面端顶部栏 / 左侧菜单的总体信息架构已批准；页面职责与路由图作为 Phase 1 skeleton 的基础。
- `SDAR-0005`：主站首页采用产品官网式结构，`/login` 承担学生与老师/辅导员的正式角色选择；低保真原型仅用于确认结构与交互，不锁定最终视觉。
- Phase 1 不接真实 API，但每个页面必须标注后续 API 串联点。
- 未批准真实学生数据、真实学校资源、SSO、生产部署或最终数据库 schema。

## 4. 官方依据

- Ant Design 支持通过 Design Token 自定义主题，包括主色、圆角、边框色等；Ant Design 5 使用 Seed Token、Map Token、Alias Token 的三层结构。
- shadcn/ui 推荐使用 CSS variables 进行 theming，提供 `background`、`foreground`、`primary`、`card`、`sidebar`、`radius` 等语义 token。
- Atlassian Design System 将 design token 作为视觉决策的统一来源，并使用基于 8px 的 spacing 体系帮助产品保持一致。
- Carbon Design System 使用 spacing token 减少布局猜测，并支持企业级产品在不同密度之间保持清晰结构。
- Material Design 提供色彩、字体、导航等成熟设计原则，可作为移动端导航和语义状态参考。
- 本审批包已补充 autoresearch 证据，详见 `.omx/specs/autoresearch-frontend-visual-design-systems/report.md`。

## 5. 推荐视觉方向

### 5.1 总体气质

关键词：

- 正式
- 可信
- 清爽
- 现代
- 轻科技感
- 高校数字服务
- AI 辅助但不喧宾夺主

避免：

- 游戏化、卡通化。
- 过度营销页。
- 大面积深蓝或紫色科技风。
- 大面积渐变、光斑、装饰圆球。
- 像临时后台、调试页面或模板拼贴。
- 像传统学校后台一样沉闷。

### 5.2 页面分层

| 区域 | 视觉方向 | 组件倾向 |
| --- | --- | --- |
| 主站首页 | 现代、正式、产品感强；产品定位和能力特性清晰；以登录入口为主 | shadcn/ui 或自定义样式 |
| 登录页 | 简洁、可信；学生/老师辅导员角色选择清晰；Demo 账号边界明确 | shadcn/ui 或自定义样式 |
| 学生端 | 轻工作台；问答优先；来源清楚 | shadcn/ui + 少量 Ant Design |
| 辅导员端 | 桌面工作台；列表、详情、状态、辅助建议 | Ant Design |
| 管理后台 | 标准后台；知识、统计、审计、重置 | Ant Design |

### 6.4 圆角与阴影 Token

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `--radius-sm` | `4px` | 标签、小按钮 |
| `--radius-md` | `6px` | 输入框、普通按钮 |
| `--radius-lg` | `8px` | 卡片、面板、弹窗 |
| `--shadow-card` | `0 1px 2px rgba(15, 23, 42, 0.06)` | 普通卡片 |
| `--shadow-popover` | `0 12px 32px rgba(15, 23, 42, 0.12)` | 浮层、菜单 |

约束：

- 工作台控件、表格、输入框默认使用 `6px` 圆角。
- 首页和学生端的大卡片或大面板可以使用 `8px` 圆角。
- 常规卡片和控件圆角不超过 `8px`。
- `16px+` 大圆角不作为默认风格；如后续需要，必须说明设计理由并重新审批。
- 不使用大圆角胶囊式卡片堆叠来制造“花哨感”。
- 阴影克制，主要用于层级，不用于装饰。

## 7. Ant Design 与 shadcn/ui 统一规则

### 7.1 Ant Design 映射

后续实现时，Ant Design `ConfigProvider` 可从全局 token 映射：

```ts
const antdTheme = {
  token: {
    colorPrimary: "var(--color-primary)",
    borderRadius: 6,
    fontSize: 14,
  },
};
```

注意：

- 这只是方向示例，不在本 SDAR 中实现。
- 具体 token 映射进入前端 skeleton 实现阶段。

### 7.2 shadcn/ui 映射

shadcn/ui 使用 CSS variables 作为主题源头，后续可将全局 token 映射到：

```css
:root {
  --background: var(--color-bg);
  --foreground: var(--color-text);
  --card: var(--color-surface);
  --card-foreground: var(--color-text);
  --primary: var(--color-primary);
  --primary-foreground: #ffffff;
  --border: var(--color-border);
  --radius: 0.5rem;
}
```

注意：

- 这只是方向示例，不在本 SDAR 中实现。
- 如果后续使用 Tailwind/shadcn，需要保持 CSS variables 开启。

## 8. 角色页面视觉说明

### 8.1 主站首页

必须体现：

- 正式产品主站。
- 产品定位、能力特性、登录入口。
- 可说明学生与老师/辅导员分别能获得的价值，但不在首页完成正式角色选择。
- Demo/模拟数据边界。
- 不展示管理后台入口。

用户交互路径：

1. 用户从学校官网链接或 Demo URL 进入首页。
2. 用户看到产品名称、Demo/模拟数据标识、一句价值说明、能力特性和登录入口。
3. 用户点击顶部“登录”或首屏“登录体验”。
4. 系统进入 `/login`。
5. 用户在登录页选择学生或老师/辅导员，并看到不同欢迎语与引导文案。
6. Demo 登录成功后进入对应工作区。
7. 管理后台通过独立链接进入，不在首页公开展示。

可使用视觉资产：

- 抽象但克制的校园数字服务背景图。
- 生成式位图资产。
- 简洁产品界面局部预览。

限制：

- 不使用真实学校照片，除非后续经过真实学校资源审批。
- 不使用纯渐变/装饰光斑作为主视觉。

### 8.2 学生端

重点：

- 问答输入清楚。
- 答案可读。
- 来源卡片显著但不干扰。
- Demo 数据标识持续存在。

用户交互路径：

1. 学生登录后进入 `/student`。
2. 页面中心展示问题输入区，并提供推荐问题、资源入口或历史会话入口。
3. 学生提交问题。
4. 系统展示回答正文、来源卡片和必要的兜底提示。
5. 学生可打开来源详情，也可查看历史会话继续提问。

### 8.3 辅导员端

重点：

- 案例列表可扫描。
- 状态、风险、建议性输出分区清楚。
- AI 输出必须标注“辅助建议”，不暗示自动决策。

### 8.4 管理后台

重点：

- 标准工作台。
- 知识资源、统计、审计、Demo 重置清晰分区。
- 高风险操作，例如 Demo 重置，必须视觉上明确危险等级。

## 10. 备选方案

| 方案 | 优点 | 缺点 | 是否推荐 |
| --- | --- | --- | --- |
| 正式高校数字服务 + 现代 AI 助手 | 可信、清爽，适合学校 Demo 和正式产品感 | 视觉需要克制，不能靠炫技制造第一印象 | 推荐 |
| 强科技感深色/渐变风 | 看起来“AI 感”强 | 容易压抑、营销化，且不适合长时间工作台使用 | 不推荐 |
| 传统后台管理系统风 | 快速、稳妥 | 主站和学生端会显得临时、缺少产品感 | 不推荐 |
| 温暖心理咨询风 | 亲和 | 可能弱化学工/知识库/管理工作台属性 | 可少量用于学生端，不作为全局方向 |

## 11. 数据边界与合规影响

- 不引入真实学生数据。
- 不引入真实学校照片或真实学校资源链接，除非后续单独审批。
- 所有展示数据必须标注 Demo/模拟。
- AI 辅助视觉元素不得暗示系统可以替代人工辅导员做决策。

## 12. 风险、缓解与回滚

| 风险 | 严重性 | 缓解方式 | 回滚方式 |
| --- | --- | --- | --- |
| Ant Design 和 shadcn/ui 风格割裂 | 高 | 使用统一 CSS variables 和设计 token | 回收自定义样式，统一 token 映射 |
| 页面过于营销化 | 中 | 首页产品化，工作台工具化 | 降低装饰，保留功能结构 |
| 色彩过于单一或过度科技风 | 中 | 使用中性底色 + 蓝/青绿/语义色组合 | 回退到 token 表 |
| 视觉过于传统或过于严肃 | 中 | 首页、登录页、学生端加入现代产品感和轻科技感；工作台保持克制 | 回到“正式可信 + 现代科技感”方向重新调 token |
| 移动端范围膨胀 | 中 | Phase 1 只做最低可用，Phase 8 polish | 延后移动增强 |
| 真实学校资产误用 | 高 | Phase 1 不使用真实学校照片/链接 | 替换为生成图或抽象资产 |
