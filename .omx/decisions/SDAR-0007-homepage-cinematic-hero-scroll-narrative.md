# SDAR-0007: 首页电影感首屏与强滚动叙事

## 1. 一句话结论

首页改造采用“强滚动叙事和电影感动效优先”的路线：顶部导航参考 Linear，首屏预留全屏视频/动效槽位，后续章节参考通义千问的滚动叙事效果，并从第一版实现就按 GSAP + ScrollTrigger 级别的动效能力设计。

## 2. 产品经理选择记录

2026-05-23，产品经理确认：

> 一开始就直接做强滚动叙事和电影感动效

处理结论：

- 不采用“先做 CSS/IntersectionObserver 保守版，再升级”的路线。
- 从首页重设计第一版开始，就按更强的滚动叙事和电影感动效组织页面。
- 因为这会引入运行时动画依赖和更高 QA 成本，所以需要本 SDAR 作为审批包。

## 3. 当前技术债边界

校徽飞翼动效不在本次首页打磨节点中继续实现。

已记录技术债：

- `.omx/logs/technical-debt-ai-counselor-demo.md`
- `TD-0001: Homepage NCHU Emblem Wing Motion Asset`

本次只做：

- 预留首屏视频/动效容器。
- 提供 poster/fallback 位置。
- 保持未来可以替换为校徽飞翼视频、Lottie、SVG+GSAP 或其他资产。

本次不做：

- 不继续手工实现三飞翼 SVG/3D 动效。
- 不承诺最终校徽动效形态。
- 不下载或使用真实学校素材，除非后续单独审批。

## 4. 参考站点拆解

### 4.1 Linear

参考内容：

- 固定顶部导航。
- 玻璃/半透明质感。
- 紧凑但高级的产品站栏目。
- 品牌、功能、资源、登录/CTA 的清晰分区。
- 首屏信息克制，强调产品定位和视觉中心。

用于本项目：

- 顶部栏目采用产品官网式导航。
- 左侧为 `NCHU AI`。
- 中间栏目建议为：`产品能力`、`使用流程`、`适用角色`、`可信边界`。
- 右侧为：`登录`、`开始体验`。
- 管理后台入口不在主站顶部公开展示。

### 4.2 通义千问

参考内容：

- 首屏大视频/动效作为第一视觉。
- hero title 和说明文字分层浮现。
- 滚动后进入大章节叙事。
- 章节内容通过 transform、opacity、blur、mask/reveal 形成滚动推进。
- 产品能力不是普通卡片堆砌，而是像“章节”一样逐步展开。

用于本项目：

- 首屏视频槽位作为未来校徽飞翼动效承载区。
- 下滑后展示 NCHU AI 的核心能力章节。
- 每个章节有大标题、简洁说明、一个产品/流程视觉区域。
- 滚动动效强调“AI 赋能校园服务”的推进感，而不是营销页卡片堆叠。

### 4.3 Apple / SpaceX

参考内容：

- Apple：极简、玻璃/液态材质、精准排版、强留白、高质量产品感。
- SpaceX：航空科技感、电影化首屏、深色场景、强视觉冲击。

用于本项目：

- 视觉风格为：`Apple 极简玻璃质感 + SpaceX 航空科技叙事 + 高校 AI 服务可信边界`。
- 避免只做深色炫技。
- 避免只做普通政务/后台风格。
- 避免装饰性光斑、无意义渐变、模板化卡片堆叠。

## 5. 推荐页面结构

### 5.1 首屏

首屏目标：

用户点击网站后第一眼看到的是完整、高级、电影化的产品入口。

结构：

1. 全屏视频/动效背景层。
2. 视频 poster/fallback。
3. `NCHU AI` 产品名。
4. 一句主价值表达。
5. 登录/开始体验 CTA。
6. 顶部 Linear-like 玻璃导航。
7. 底部滚动提示。

首屏行为：

- 页面进入时，视频/动效层先出现。
- 顶部导航从顶部向下渐进呈现。
- 标题和说明文字使用轻微 blur/translate/reveal。
- 完成首屏进入后，用户滚轮下滑进入章节叙事。

### 5.2 滚动章节

初始章节建议：

| 顺序 | 章节 | 作用 |
| --- | --- | --- |
| 1 | AI 赋能校园咨询 | 说明学生端问答、来源、边界 |
| 2 | 辅导员工作辅助 | 说明案例、状态、辅助建议 |
| 3 | 知识运维与审计 | 说明知识资源、重置、审计 |
| 4 | 可信 Demo 边界 | 说明模拟数据、非生产 SSO、非真实学生数据 |

章节表现：

- 每个章节接近一屏或半屏以上。
- 内容不要只是普通 Card。
- 使用大标题、短文案、视觉预览区、局部动效。
- 滚动时产生渐入、位移、遮罩、缩放、固定/释放等效果。

## 6. 技术方案

### 6.1 已确认方向

采用强滚动叙事路线。

推荐实现：

- `GSAP`
- `@gsap/react`
- `ScrollTrigger`

原因：

- 更接近通义千问这类滚动叙事效果。
- 适合多个 section 的 timeline、pin、scrub、stagger、mask/reveal。
- 当前已经安装了 GSAP 相关开发 skills，可以降低实现偏差。
- React 中可以用 `@gsap/react` 做生命周期清理，避免动画残留。

### 6.2 需要新增的运行时依赖

如果本 SDAR 审批通过，进入实现节点时建议安装：

```text
gsap
@gsap/react
```

暂不安装：

- `motion`
- `lenis`
- `three`
- `@react-three/fiber`
- `@lottiefiles/dotlottie-react`
- `lottie-react`
- `@rive-app/react-canvas`

理由：

- 本次主要是强滚动叙事，不是 3D 引擎。
- 校徽飞翼动效已进入技术债，不需要马上安装 Lottie/Rive。
- 平滑滚动库 `lenis` 可能影响可访问性和滚动调试，先不引入。

### 6.3 实现边界

本节点允许：

- 改造首页首屏。
- 改造顶部导航样式和入场动效。
- 新增滚动章节结构。
- 使用模拟/占位内容表达产品能力。
- 增加本地占位 video/poster/fallback 容器。

本节点不允许：

- 改登录逻辑和角色路由契约。
- 改后端 API。
- 引入真实学生数据。
- 引入生产 SSO。
- 引入真实学校资源。
- 继续实现校徽飞翼终版动效。

## 7. 设计约束

必须保持：

- 桌面端优先。
- 移动端可读、可用、不严重溢出，但完整 polish 可后置。
- 顶部导航不能遮挡关键信息。
- 首屏视频没有加载完成时必须有静态 fallback。
- 用户开启减少动态效果时，动效必须降级。
- 不能复制 Linear、通义、Apple、SpaceX 的品牌素材和文案。
- 最终 Demo 仍需明确模拟数据边界。

## 8. 验证方式

实现节点完成后至少验证：

- `npm run build`
- `npm run lint`
- 桌面截图。
- 移动截图。
- 首页首屏视频槽位存在。
- 顶部导航入场可见。
- 下滑章节触发滚动动效。
- `prefers-reduced-motion` 下可降级。
- 登录入口仍能进入 `/login`。
- 角色登录和路由不被破坏。

## 9. 风险与缓解

| 风险 | 严重性 | 缓解 |
| --- | --- | --- |
| 动效过强导致页面像炫技站 | 中 | 每个动效必须服务章节叙事，不做无意义装饰 |
| GSAP 引入后生命周期清理不当 | 中 | 使用 `@gsap/react`，组件卸载时清理 timeline/ScrollTrigger |
| 首屏视频资源过大 | 中 | 先用占位容器和 poster，真实视频后续单独压缩验收 |
| 移动端滚动体验差 | 中 | 桌面端优先，移动端降级为简洁 reveal |
| 影响登录路由 | 高 | 首页改造不触碰 auth/session 逻辑 |
| 校徽动效继续消耗时间 | 中 | 保持 TD-0001 技术债，当前只留槽位 |

## 10. 回滚方式

如果效果不达标或依赖引入失败：

- 回滚首页动效实现。
- 保留当前 Phase 2 登录/角色路由成果。
- 首页可回退到当前静态首屏。
- 删除 `gsap` / `@gsap/react` 依赖和相关组件。
- 保留技术债记录，不恢复旧飞翼原型为生产实现。

### 产品经理逐条回复与 Codex 回应

#### 11.1 是否批准实现节点引入 `gsap` 和 `@gsap/react`？

> 产品经理回复：

**Codex 回应**

确认采纳。实现节点允许安装：

```text
gsap
@gsap/react
```

实现时必须使用已安装的 GSAP 系列 skills：

- `gsap-core`

- `gsap-performance`
- `gsap-plugins`

约束：

- 只为首页强滚动叙事引入 GSAP。
- 不借此引入 Three.js、Rive、Lottie、Lenis 或其他动画依赖。
- React 组件中必须使用 GSAP React 生命周期清理方式，避免滚动触发器残留。

#### 11.2 首屏整体更偏深色电影感，还是浅色玻璃科技感？

> 产品经理回复：
>
> 深色电影感；交互按钮，顶部导航栏目等按钮用玻璃科技感风格  
> 还有，动效制作的时候一定要使用gsap系列skills

**Codex 回应**

确认采纳。

首屏视觉方向调整为：

- 整体氛围：深色电影感。
- 材质细节：顶部导航、按钮、CTA、局部状态标签使用玻璃科技感。
- 叙事气质：航空科技 + AI 赋能 + 高校可信服务。
- 禁止方向：不能做成普通深色后台、纯炫技科幻页、无意义光斑堆叠页。

实现要求：

- 首屏背景以深色视频/动效区域为主。
- 导航采用 Linear-like 固定玻璃导航。
- 交互按钮使用玻璃态边框、轻微高光、明确 hover/focus 状态。
- 动效使用 GSAP 系列 skills 设计和实现。

#### 11.3 首屏视频槽位第一版如何处理？

> 产品经理回复：
>
> 可以，但是我想听听你的方案，你是寻找一个视频先插入进去吗？还是说什么？

**Codex 回应**

我的建议是：**第一版不要从网上随便找一个视频插进去**。

原因：

- 随机公开视频容易有版权、品牌不匹配、加载不稳定、国内访问不稳定的问题。
- 真实视频如果风格不准，会把首页审美带偏，后续反而更难调整。
- 当前校徽飞翼视频已经进入技术债，第一版的目标应该是把“视频槽位、导航入场、滚动叙事、整体质感”跑通，而不是临时找素材填满。

推荐方案：

第一版使用 `GSAP + CSS/SVG` 做一个“抽象航空 AI 占位动效层”，视觉上像视频，但本质是可控的前端动效。

占位层内容建议：

- 深色电影背景。
- 细微航空网格 / 航迹线 / 扫描线。
- 低透明度粒子或数据流。
- 模拟飞行路径从左下向右上推进，但不继续实现校徽飞翼。
- 中央保留未来视频/校徽动效的视觉重心。
- 提供静态 poster/fallback。

这样做的优点：

- 不依赖外部视频资源。
- 没有版权风险。
- 和页面滚动叙事可以统一由 GSAP 控制。
- 后续你提供最终视频后，可以直接把该占位层替换成 `<video>` 或 Lottie。
- 视觉方向可以快速调，不会被临时素材限制。

备选方案：

| 方案 | 说明 | 是否推荐 |
| --- | --- | --- |
| A. GSAP/CSS/SVG 代码生成占位动效 | 前端直接制作抽象航空 AI 场景 | 推荐 |
| B. 找一个网络公开视频临时插入 | 快，但版权、风格、访问稳定性都有风险 | 不推荐 |
| C. 生成一个临时视频资产 | 效果可能更像视频，但需要额外资产生成和验收 | 可作为后续增强 |

当前处理结论：

- 第一版采用方案 A。
- 不从网络寻找随机视频。
- 保留未来替换成真实视频的结构。
