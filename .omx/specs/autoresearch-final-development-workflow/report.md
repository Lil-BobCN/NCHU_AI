# 最终开发流程：产品想法到可验证交付

日期：2026-05-19
状态：accepted with 2026-05-19 workflow amendments
适用范围：当前 AI 辅导员 Demo，以及后续类似产品项目

## 1. 一句话结论

我们最终采用一套“**先澄清、再调研、再审批、再原型、再拆任务、再实现验证**”的开发流程。

它的核心不是让我一次性写很多代码，而是把你的产品想法一步步翻译成可开发、可审批、可验证的小任务：

```text
想法/文档
-> 需求澄清
-> 调研与方案
-> 多方案思想对齐
-> 审批包/SDAR
-> 产品与前端骨架
-> 用户交互顺序和低保真原型
-> 页面结构、路由、API、数据契约
-> 小任务实现
-> 测试验证
-> 验收记录与路径图更新
```

## 4.1 新增流程治理规则

### 多方案思想对齐

如果一个任务存在多个可替代方案、实现路径或产品表达方式，Codex 不能直接替产品经理拍板。正确流程是：

```text
发现多方案
-> 列出选项、优缺点、影响范围、风险和回滚
-> 给出 Codex 推荐
-> 与产品经理思想对齐
-> 必要时进入 Deep Interview 式澄清
-> 再进入审批或实现
```

该规则适用于技术路线、UI/交互结构、页面排版、数据契约、开源组件、后端边界、部署方式等会影响长期方向的选择。

### 开发路径结构图

项目日志之外，必须维护一份“开发路径结构图”。它的作用是让产品经理一眼看到任务节点、当前进度、审批门控和下一步。

当前项目路径图位置：

```text
.omx/logs/development-path-structure-ai-counselor-demo.md
```

每完成一个任务节点、审批包、验证或验收动作，都应同步更新项目日志和路径图。

## 3. 我记得的既定开发原则

你是产品经理，我是主程序员。

每个关键技术选型、UI 结构、数据库结构、数据契约、开源组件引入、真实学校资源、SSO、真实学生数据、公开部署和生产安全边界，都必须先给你看审批包，再继续。

我不能一次性闷头做很多任务。正确方式是先拆成小任务节点，每个节点说明范围、影响、风险、验收和回滚。

前端最终必须和后端 FastAPI API 串联。Phase 1 可以先不接真实 API，但每个页面要写清楚未来要接什么 API。

桌面端优先，移动端要尽量兼顾，但不能为了移动端牺牲桌面端工作台质量。

前端技术栈已批准为 React + TypeScript + Vite + Ant Design / shadcn-ui。

后端边界已批准为 FastAPI 作为主业务/API 权威；后续 RAG/Agent 工作流采用 LangGraph 为核心，LangChain 做有限工具集成，LangSmith 不默认进入真实学校数据链路。

## 4. 最终流程总览

### Step 0：需求入口与上下文归档

目标：
把你的想法、上传文档、现有代码和历史计划先归档，不直接开写。

我需要做：
- 读取已有 PRD、开发计划、SDAR、代码现状。
- 记录当前目标、约束、已确认事项、未知问题。
- 如果需求不清楚，进入 `$deep-interview`。

产物：
- `.omx/context/...md`
- 必要时补充 `.omx/interviews/...md`

停止条件：
核心目标、非目标、审批边界不清楚时，不进入开发。

### Step 1：需求澄清

目标：
把模糊想法变成能判断范围的需求。

适用情况：
- 你说“我大概想做一个东西”。
- 页面、功能、角色、数据、交互还不清楚。
- 你明确要求不要我自行假设。

推荐工具：
- `$deep-interview`

产物：
- 需求澄清记录。
- 目标、非目标、约束、验收方式。

停止条件：
如果还有关键歧义，继续问；不进入实现。

### Step 2：调研与方案比较

目标：
当涉及技术方案、开源组件、设计模式、行业做法时，先调研，再推荐。

适用情况：
- 是否引入某个框架。
- 选择前端组件库。
- 选择 RAG / Agent / 可观测方案。
- 参考成熟网站、设计系统、交互模式。

推荐工具：
- `$autoresearch`

产物：
- `.omx/specs/autoresearch-*/report.md`
- 推荐方案、备选方案、为什么推荐、风险和验证方法。

停止条件：
没有调研证据时，不做关键选型。

### Step 3：审批包 / SDAR

目标：
让你像产品经理一样集中审批关键决策。

适用情况：
- 技术选型。
- UI 结构。
- 数据库结构。
- 数据契约。
- 开源组件。
- 安全/部署/真实数据边界。

产物：
- `.omx/decisions/SDAR-xxxx-*.md`

每个审批包必须包含：
- 推荐方案。
- 备选方案。
- 为什么推荐。
- 影响范围。
- 风险。
- 回滚方式。
- 验收方式。
- 需要你确认的问题。
- 你的审批结果和日期。

停止条件：
未审批的关键决策，不进入实现。

### Step 4：产品骨架与功能模块

目标：
先确定产品由哪些功能模块组成，而不是一上来写页面和代码。

本项目当前确定：
- 网站入口模块。
- 登录认证模块。
- 学生咨询模块。
- 辅导员工作台模块。
- 管理后台模块。
- 知识资源模块。
- 统计与审计模块。

产物：
- 前端页面结构文档。
- 模块清单。
- 每个模块的目标、边界和优先级。

停止条件：
模块职责不清楚时，不进入页面设计。

### Step 5：用户交互顺序

目标：
模拟用户怎么一步一步使用网站，用交互顺序反推页面排版。

例子：

```text
用户打开网站
-> 点击“我是学生”
-> 进入登录页
-> 登录成功
-> 进入学生工作台
-> 输入问题
-> 看到回答和来源卡片
```

为什么重要：
- 用户第一步要做什么，决定主按钮放哪里。
- 用户下一步要看什么，决定主内容区和侧边栏怎么排。
- 用户需要确认什么，决定弹窗、提示、状态卡片怎么设计。

产物：
- 用户流程清单。
- 交互顺序说明。
- 必要时配低保真原型。

当前原型：
- `.omx/prototypes/frontend-interaction-flow-demo.html`

停止条件：
核心用户流程没走通时，不进入正式视觉设计。

### Step 6：低保真交互原型

目标：
先用简单页面让你“看见流程”，不纠结正式视觉。

原型应该表达：
- 用户点击什么。
- 跳到哪个页面。
- 当前页面属于哪个模块。
- 为什么这样排版。
- 后续要接哪个 API。

原型不做：
- 不等于正式 React 实现。
- 不要求最终视觉精细。
- 不引入新的正式依赖。

产物：
- `.omx/prototypes/*.html`

停止条件：
你没有确认核心交互顺序前，不进入正式页面结构定稿。

### Step 7：页面结构、路由图、API 串联点

目标：
把原型转成工程可用的开发说明。

每个页面必须写清：
- 所属功能模块。
- 页面名称。
- 路由 URL。
- 页面目的。
- 主要区域。
- 按钮、表格、卡片、弹窗。
- 空状态、加载态、错误态、无权限态。
- Phase 1 是否只是 skeleton。
- 后续要接的 FastAPI API。
- 技术债或后续确认项。

产物：
- 页面结构文档。
- 路由/API 映射表。

停止条件：
如果页面没有 API 串联点或数据边界说明，不能进入 API 连接型开发。

### Step 8：视觉方向与 Design Token

目标：
在结构和交互确认后，再确定“好不好看、像什么风格”。

内容包括：
- 产品气质。
- 颜色。
- 字体。
- 间距。
- 圆角。
- 阴影。
- 卡片密度。
- 按钮和表单风格。
- Ant Design 与 shadcn/ui 的样式统一方式。

为什么放在这一步：
如果先定视觉，很容易做出好看的页面，但流程不对、信息层级不对、后端串联不清楚。

产物：
- `SDAR-0004` 或后续视觉决策文档。
- Design token 表。

停止条件：
视觉不能覆盖未确认的页面结构和交互逻辑。

### Step 9：数据契约与后端边界

目标：
前后端正式串联前，先定数据结构和责任边界。

必须明确：
- `User/Role`
- `KnowledgeResource`
- `Conversation`
- `CounselorCase`
- `AuditEvent`
- `StatsSnapshot`

每个契约要写：
- 字段。
- 类型意图。
- 谁读。
- 谁写。
- 是 Demo 数据还是真实数据。
- 是否持久化。
- 是否需要审计。

后端原则：
- FastAPI 是业务/API 权威。
- Phase 1 可不接真实 API。
- 最终 Demo 必须完成前后端闭环。
- RAG/Agent 进入后续阶段时，LangGraph 工作流不能绕过 FastAPI 业务边界。

产物：
- 数据契约审批包。
- API contract。

停止条件：
未批准数据契约前，不实现 API 连接型 UI。

### Step 10：小任务节点拆解

目标：
把开发拆成可以单独做、单独验收、单独回滚的小节点。

每个任务节点必须写清：
- 目标。
- 范围。
- 涉及文件。
- 不做什么。
- 验收方式。
- 回滚方式。

推荐粒度：
- 一个页面骨架。
- 一个 API 接口。
- 一个数据契约。
- 一个登录流程。
- 一个表格/表单行为。
- 一条用户流程。

不推荐：
- “一次性把前端做完”。
- “一次性把后台全部做完”。
- “先写代码后补文档”。

产物：
- phase task nodes。
- 实现计划。

### Step 11：实现

目标：
只实现已批准、已拆分、可验证的小任务。

执行原则：
- 小改动。
- 贴合现有代码结构。
- 不引入未审批依赖。
- 不扩大范围。
- 不碰无关文件。
- 不把 Demo 数据伪装成真实数据。

推荐工具：
- 简单单点任务：solo。
- 需要持续修复和验证：`$ralph`。
- 多个明确分工、互不冲突的任务：`$team`。

停止条件：
实现中发现新的技术、UI、数据、数据库、组件或安全决策，停止并补审批包。

### Step 12：验证

目标：
证明这次改动真的工作，而不是只凭感觉完成。

按任务选择验证：
- 前端：类型检查、构建、页面打开、浏览器截图、交互检查、响应式检查。
- 后端：单元测试、API 测试、健康检查、权限检查。
- 数据库：迁移、回滚、重启后数据保留。
- RAG：来源引用、兜底、检索质量、可恢复到确定性适配器。
- 部署：Docker Compose、health/readiness、Demo 脚本。

产物：
- 测试输出摘要。
- 截图或 smoke evidence。
- 验收记录。

停止条件：
未验证的功能不能称为完成。

### Step 13：验收、日志与回滚记录

目标：
每个阶段结束时，留下可追踪记录。

产物：
- `.omx/logs/...md`
- phase acceptance summary。
- 风险列表。
- 已知问题。
- 回滚说明。
- 下一步建议。

验收内容：
- 做了什么。
- 没做什么。
- 如何验证。
- 哪些风险仍存在。
- 是否需要你审批进入下一步。

## 5. 前端专用流程

以后前端规划固定采用：

```text
功能模块
-> 用户交互顺序
-> 低保真交互原型
-> 页面结构文档
-> 路由/API 映射
-> 视觉方向和 design token
-> React skeleton
-> API 串联
-> 响应式 QA
```

这解决的是：
- 你脑子里的模糊想法如何变成页面。
- 页面为什么这样排版。
- 我开发时应该先做哪一页。
- 后端 API 最终接在哪里。

## 6. 后端专用流程

后端规划固定采用：

```text
业务目标
-> API 边界
-> 数据契约
-> 服务层职责
-> 存储选择
-> 鉴权与审计
-> Demo in-memory 实现
-> 持久化迁移
-> RAG/Agent 工作流
-> 可观测与评测
```

当前项目后端关键原则：
- FastAPI 保持主 API 和业务编排边界。
- LangGraph 后续作为 RAG/Agent workflow 核心。
- LangChain 只做有限工具集成。
- LangSmith 不默认进入真实学校数据链路。
- PostgreSQL、Redis、MinIO、Milvus、Redis Stream 都必须按阶段审批后接入。

## 7. 什么时候用哪个工作流

| 场景 | 使用方式 |
| --- | --- |
| 需求不清楚 | `$deep-interview` |
| 需要查资料、比较方案、找成熟案例 | `$autoresearch` |
| 需要制定阶段计划或技术路线 | `$ralplan` |
| 已批准的小任务需要持续做到完成 | `$ralph` |
| 多个独立任务可并行开发 | `$team` |
| 前端视觉与截图反复对齐 | `$visual-ralph` / `$visual-verdict` |
| 代码完成后需要质量检查 | `$code-review` / `$ultraqa` |

## 8. 关键停止规则

遇到以下情况必须停下来，不继续实现：

- 新技术选型未审批。
- UI 结构变化未审批。
- 数据库结构未审批。
- 数据契约新增字段或行为未审批。
- 开源组件引入未审批。
- 涉及真实学生数据。
- 涉及真实学校资源。
- 涉及 SSO。
- 涉及公开部署或生产安全边界。
- 前端页面没有明确后续 API 串联点。
- 用户流程和页面结构还没有通过原型或文档确认。

## 9. 可复用模板

以后启动任何新项目，可以按这个顺序问和做：

1. 这个产品解决什么问题？
2. 谁使用它？
3. 用户第一步做什么，下一步做什么？
4. 产品由哪些功能模块组成？
5. 每个模块有哪些页面？
6. 每个页面有哪些状态、按钮、表格、弹窗？
7. 每个页面未来接哪个 API？
8. 数据由谁读、谁写、是否持久化、是否审计？
9. 哪些技术和组件需要审批？
10. 哪个最小任务节点可以先实现并验证？

## 10. 建议写入主计划的流程补充

建议把以下规则补进总体开发方案：

```text
前端开发进入正式 React 实现前，必须完成：
1. 功能模块划分；
2. 用户交互顺序；
3. 低保真交互原型；
4. 页面结构文档；
5. 路由/API 映射；
6. 视觉方向和 design token 审批。

Phase 1 可以不连接真实后端 API，但每个页面必须明确最终 FastAPI API 串联点。
```

## 11. 当前结论

你提出的新流程是合理的，并且应该成为以后所有前端开发的标准流程。

最终流程不是“先做 UI”，也不是“先写代码”，而是：

```text
先把产品怎么用讲清楚
再把页面怎么组织讲清楚
再把后端怎么串讲清楚
最后才进入正式开发
```
