# AI Counselor System Positioning Research

Date: 2026-05-15

## Research Question

根据当前项目文件、学校登录系统公开信息、同类高校案例和未来规划，判断本系统更应定位为哪一类产品，并列出后续筛选时需要考虑的条件。

## Local Project Evidence

当前仓库和项目文档显示：

- 当前已完成/正在收敛的正式技术底座是 `backend/` FastAPI + PostgreSQL + Redis + MinIO + Milvus + Docker Compose。
- `README.md` 和 `docs/FASTAPI-SYSTEM.md` 明确说明技术阶段一只暴露 `/api/v1/health` 和 `/api/v1/readiness`，不暴露 auth、chat、RAG 问答、后台、通知、大屏和前端业务页面。
- 原始需求清单将第一阶段定义为“基础轻量化阶段”，核心是校本知识库、基础大模型对话、7x24 答疑、官方依据与办事路径、资源检索直达、知识库后台、三级权限、日志统计、通知文案和多终端接入。
- 开发计划表的后续阶段包含文档解析、切片向量化、检索精排、多轮记忆、业务 API、鉴权、前端双端适配、测试上线。
- RAG 技术方案强调轻量适配、模块解耦、数据分层、可观测和成本可控，技术路线是 FastAPI + LangGraph + PostgreSQL/Redis + Milvus + MinIO。
- NCHU SSO 公开调研显示学校登录更适合 FastAPI 内实现 `NchuSsoAdapter`，而不是把 Casdoor 作为当前主认证依赖。

## External Evidence

- 教育部第43号令将辅导员定位为大学生思想政治教育、日常管理、组织实施与指导的重要角色，工作覆盖思想理论教育、党团班级建设、学风建设、日常事务、心理健康、网络思政、危机应对、职业规划与就业、研究等方面。
- 《生成式人工智能服务管理暂行办法》要求生成式 AI 服务遵守合法合规、核心价值观、准确可靠、个人信息保护等原则，这支持本项目坚持“AI 辅助、人工主导、不替代辅导员”的边界。
- 重庆大学 AI 辅导员“润欣”公开案例显示成熟形态会从知识库问答扩展到跨平台业务办理、学业/心理/就业等综合服务、辅导员工作台与决策辅助。
- 浙江工商职业技术学院“AI 辅导员”案例显示早期价值重点是 7x24 高频事务问答、校务咨询、知识库维护、重复答疑减负和人工转接。
- 河北软件职业技术学院“云蹊老师”案例显示可用“制度智库 + 问题案例库 + 统一身份认证”作为轻量化落地路径，并通过需求热点分析反哺辅导员工作。

## Positioning Conclusion

本项目当前最适合定位为：

**校本知识库驱动的学生事务自助问答系统 + 辅导员工作辅助平台。**

不是单纯聊天机器人，也不是第一版就做完整智慧学工、一网通办、心理风险闭环或全域学生画像系统。

推荐主线：

1. 先做学生高频事务自助问答和校内资源直达。
2. 同时建设知识库后台和基础权限，保证内容可维护、可追溯。
3. 再做辅导员工作台能力，例如高频问题分析、通知文案、谈话提纲、台账辅助。
4. 最后在学校授权和数据安全边界明确后，再做画像、预警、跨系统业务办理和大屏决策。

## Candidate Product Shapes

### A. 学生自助问答助手

作用：解决学生高频咨询、政策流程查询、办事入口寻找。

适合优先级：P0/P1 主入口。

理由：需求清单第一阶段最明确，技术方案 RAG 路线直接服务这个目标，落地快、风险低、减负明显。

### B. 校内资源一站式检索直达

作用：返回官方文件、表格、制度依据、办理路径、跳转链接。

适合优先级：P1 必做。

理由：能显著降低 AI 胡编风险，让回答可追溯，也符合“官方依据”验收要求。

### C. 校本知识库运营后台

作用：让管理员/部门维护文档、问答、分类、标签、错误纠正和日志。

适合优先级：P1 必做或 P0 后半段。

理由：没有后台，知识库无法持续更新；没有纠错和日志，RAG 质量无法迭代。

### D. 辅导员工作台助手

作用：生成通知、谈话提纲、工作总结、帮扶记录初稿，分析高频问题。

适合优先级：P1/P2。

理由：价值明显，但必须建立在问答、知识库、用户身份和日志基础上。

### E. 权限、审计和统计看板

作用：支撑校级管理员、院系辅导员、学生用户的权限隔离和行为追溯。

适合优先级：P0/P1 基础能力。

理由：高校场景不能没有权限边界和日志，尤其涉及对话记录、知识库变更和后台操作。

### F. 学生画像与风险预警

作用：接入学业、考勤、奖惩、资助、心理等数据后，做提醒、标记和人工介入流转。

适合优先级：P2/P3。

理由：价值高但合规、数据授权、误判责任和人工闭环要求高，不适合当前第一版主线。

### G. 业务办理 / 一网通办智能体

作用：通过问答直接办请假、证明、补办、申请等流程。

适合优先级：P2/P3。

理由：需要对接校内业务系统和流程权限。当前项目先做“问清楚、找得到、能跳转”，再做“直接代办”。

### H. 数字人 / 多模态思政交互

作用：视频、语音、形象化交互、沉浸式思政内容。

适合优先级：P3。

理由：展示效果强，但对核心业务闭环帮助不如知识库、权限、问答和后台。

## Recommended P0/P1 Product Boundary

### P0: 后端与身份基础

- FastAPI 正式后端。
- NCHU SSO Adapter + 本地开发登录。
- 用户表、角色表、组织/学院/班级基础字段。
- PostgreSQL 业务数据模型和迁移。
- Redis 会话/临时状态。
- MinIO 原始文档存储。
- Milvus 向量索引基础。
- 健康检查、readiness、smoke、日志。

### P1: 可用的轻量业务闭环

- 学生端问答 API。
- 多轮会话与历史记录。
- RAG 检索、重排、来源引用。
- 校内资源链接返回。
- 知识库文档上传、解析状态、入库、重建索引。
- 管理后台的知识库维护、错误纠正、日志检索。
- 基础权限：校级管理员、院系辅导员、学生。
- 基础统计：咨询量、问题分类、高频问题、失败/无答案问题。

## Non-goals For First Business Version

- 不做心理诊断。
- 不做自动奖惩、处分、资助资格最终决策。
- 不直接接入全量敏感学生数据。
- 不做完整风险闭环预警系统。
- 不做全域一网通办代办。
- 不做数字人大屏优先。
- 不把 AI 回答当作最终政策解释，以官方文件和人工审核为准。

## Decision Points For User Screening

1. 第一版主入口是否优先面向学生，而不是辅导员后台。
2. 第一版是否必须包含知识库后台，否则只能做演示型问答。
3. 第一版是否接受“只返回办理路径/链接”，暂不直接办理业务。
4. 是否把 NCHU SSO 作为正式登录方向，同时保留本地开发登录。
5. 是否将心理、画像、风险预警明确后置到 P2/P3。
6. 是否将通知文案、谈话提纲、台账辅助列为辅导员工作台的 P1/P2 功能。
7. 是否要求所有回答必须带来源引用或无法回答时兜底。
8. 是否优先支持官网/H5 入口，企业微信/公众号后置。
